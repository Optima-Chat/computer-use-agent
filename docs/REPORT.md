## 架构

```
┌──────────────────────────────────────────────────────┐
│  Host (Bun + TypeScript)                             │
│                                                      │
│  run.ts ──▶ agent.ts ──▶ Anthropic API (Beta)        │
│                ↕                                     │
│             executor.ts ──▶ docker exec ──▶ Container │
│                                    │                 │
│                        Xvfb + Fluxbox + Firefox      │
│                        xdotool / scrot               │
└──────────────────────────────────────────────────────┘
```

Agent Loop: 截图 → Claude 分析 → 执行工具(点击/输入/bash) → 循环，直到模型回复 `TASK_COMPLETE` 或达到 30 轮上限。

## 关键发现

### 1. 纯终端任务效率极高

Task 02（创建目录结构）仅用 **3 轮迭代、8.4 秒**，模型直接用 bash 一条命令搞定，不需要截图交互。这类任务 Computer Use 本质上退化为 bash 调用。

### 2. GUI 操作需要大量迭代

Task 03（Mousepad 编辑）和 Task 06（查找替换）分别需要 12 和 22 轮迭代（Opus 768x576）。每轮都要截图 → 分析 → 操作，截图是 base64 PNG，token 消耗巨大。

### 3. Opus vs Sonnet 策略差异

**Task 04 对比**（Sonnet FAIL vs Opus PASS）:

- **Sonnet**: 直接看截图操作，填了表单但 submit 后陷入循环，未完成后续 `xdotool getwindowname` 保存步骤
- **Opus**: 先执行 `cat form.html` 读取 HTML 源码理解页面结构，再有策略地填写，最终完成全流程

Opus 展现了**先理解再操作**的策略，而非纯视觉试错。

### 4. 降低分辨率显著节省 token

将分辨率从 1024x768 降到 768x576 后：
- 每张截图 token: ~1049 → ~590（减少 44%）
- Opus 7 任务总 token: 324K（远低于 Sonnet 1024x768 的 1.23M）
- 模型在低分辨率下依然能准确定位 GUI 元素

| 分辨率 | 每截图 tokens | 公式 |
|--------|--------------|------|
| 1024x768 | ~1,049 | (1024×768)/750 |
| 768x576 | ~590 | (768×576)/750 |

注意：token 计算纯按像素，与图片格式/压缩无关。

### 5. Token 消耗分析

| 场景 | Tokens | 估算成本 |
|------|--------|----------|
| Sonnet 4.6 × 7 任务 (1024x768) | ~1.23M | $4-6 |
| Opus 4.6 × 7 任务 (768x576) | ~324K | $5-8 |

### 6. 推理 vs VM 执行延迟分解

通过在 agent loop 中对每轮迭代进行 profiling，得到实际延迟分布（Opus 4.6, 768x576）：

```
AGGREGATE
├─ Inference:  339.5s (91%)
├─ VM Total:    34.6s (9%)
│  ├─ Screenshots:  5.6s (17 captures, 1004KB total)
│  ├─ GUI Actions: 10.9s
│  └─ Bash:        18.1s
└─ Total:      374.1s

Avg screenshot: 327ms capture, 59KB/frame
```

| 任务 | 推理时间 | VM 时间 | VM 占比 | 截图数 |
|------|---------|---------|---------|--------|
| 01 创建文件 | 10.1s | 0.6s | 6% | 1 |
| 02 目录结构 | 6.8s | 0.6s | 8% | 0 |
| 03 GUI 编辑 | 57.2s | 5.7s | 9% | 2 |
| 04 浏览器表单 | 113.9s | 15.1s | 12% | 5 |
| 05 终端管道 | 9.0s | 0.6s | 7% | 0 |
| 06 查找替换 | 102.8s | 8.4s | 8% | 6 |
| 07 多步工作流 | 39.6s | 3.7s | 9% | 3 |

**关键发现**: 使用远程 API 调用时，**推理占 91%，VM 仅占 9%**。这与预期的 30-50% VM 开销差异很大——原因是 API 往返延迟（网络 + 排队 + GPU）远大于本地 VM 操作。但在**本地部署场景**（自托管模型），VM 侧占比会显著上升。

### 7. 失败模式

Sonnet 在 Task 04 的失败模式：
1. 正确打开 Firefox 并导航到 form.html
2. 正确填写了 Name 和 Email
3. Department 下拉框操作出现问题（选择 Engineering 但 select 交互不稳定）
4. 反复尝试提交，消耗完 30 轮迭代
5. **根因**: 缺乏对 HTML 结构的理解，纯靠视觉定位不够精确

## 结论

1. **Computer Use 已可用于真实桌面自动化** — Sonnet 6/7 (86%), Opus 7/7 (100%)
2. **纯终端任务无需 Computer Use** — bash 工具足矣，速度快 10x，成本低 100x
3. **GUI 任务的价值在于无 API 的应用** — 填表单、操作编辑器等没有 CLI/API 的场景
4. **Opus 在复杂 GUI 任务上显著优于 Sonnet** — 策略性更强，能主动获取上下文而非纯视觉试错
5. **降低分辨率是有效的优化手段** — 768x576 vs 1024x768 节省 44% 截图 token，且不影响准确性
6. **成本可控** — 降分辨率后 Opus 全跑 7 任务仅 ~324K tokens，适合实际应用
